\# Documentation Guidelines



\## Purpose



This document defines how documentation should be organized throughout the project.



\---



\## General Rules



Each document should contain:



\- Purpose

\- Scope

\- Responsibilities (if applicable)

\- Main Content

\- References



\---



\## Writing Principles



Documentation should:



\- Focus on one topic

\- Avoid duplicated information

\- Reference related documents instead of repeating them

\- Use clear and concise language



\---



\## Cross References



When referring to another document, reference its path.



Example:



```

docs/02-domain-model/order/order.md

```



\---



\## File Organization



Each directory should contain:



\- README (optional)

\- Overview document (when appropriate)

\- Topic-specific documents



\---



\## Markdown Rules



\- Use ATX headings (`#`)

\- Use fenced code blocks

\- Use tables only when appropriate

\- Keep paragraphs short



\---



\## Updates



Documentation should be updated together with architecture or business changes.



Documentation must never lag behind implementation.
