# Purpose

เอกสารนี้กำหนด "ปรัชญาการโต้ตอบกับผู้ใช้" ของ JLek Platform

ไม่ได้กำหนดหน้าตา UI แต่กำหนดว่าระบบต้อง "รู้สึก" อย่างไรเมื่อผู้ใช้ใช้งาน

------------------------------------------------------------------------

# Vision

> ผู้ใช้ไม่ควรเรียนรู้ระบบ
>
> ระบบควรเรียนรู้วิธีการทำงานของผู้ใช้

------------------------------------------------------------------------

# Core Principles

## 1. Never Surprise the User

ทุกการกระทำต้องคาดเดาได้ ผู้ใช้ควรรู้ว่าจะเกิดอะไรขึ้นหลังการกดปุ่ม

## 2. Immediate Feedback

ทุก Action ต้องได้รับการตอบสนองทันที เช่น Loading, Progress, Saving หรือ Success

## 3. One Action → One Clear Result

หนึ่งการกระทำควรมีผลลัพธ์หลักที่ชัดเจน หากมีงานเบื้องหลังหลายอย่าง ต้องสื่อสารให้ผู้ใช้เข้าใจ

## 4. Preserve User Work

ห้ามทำให้ข้อมูลสูญหายโดยไม่จำเป็น รองรับ Draft, Auto Save และ Undo เมื่อเหมาะสม

## 5. Explain, Don't Blame

ข้อความผิดพลาดต้องอธิบาย - เกิดอะไรขึ้น - สาเหตุที่เป็นไปได้ - สิ่งที่ผู้ใช้ควรทำต่อ

## 6. Speak Human

ใช้ภาษาธุรกิจ ไม่ใช้ศัพท์เทคนิค เช่น - "กำลังพิมพ์ใบเสร็จ" - "ส่งเข้าครัวแล้ว" -
"บันทึกสำเร็จ"

## 7. Perceived Performance Matters

ความเร็วที่ผู้ใช้ "รับรู้" สำคัญกว่าความเร็วของเครื่อง ใช้ Animation และ Progress
อย่างพอดีเพื่อสร้างความมั่นใจ

## 8. Focus on the Current Task

ทุกหน้าต้องตอบคำถามว่า "ตอนนี้ผู้ใช้กำลังทำอะไร" แล้วเน้นเฉพาะข้อมูลที่จำเป็นต่อการตัดสินใจ

## 9. Mistakes Are Normal

ผู้ใช้กดผิดได้ ระบบต้องช่วยแก้ไข ไม่ใช่ลงโทษ

## 10. Respect Time

ทุกคลิกมีต้นทุน ลดขั้นตอนที่ไม่จำเป็น และหลีกเลี่ยง Dialog ที่ขัดจังหวะงาน

# Interaction States

ทุกการโต้ตอบควรอยู่ในสถานะใดสถานะหนึ่ง

-   Idle
-   Processing
-   Success
-   Warning
-   Error
-   Offline

ผู้ใช้ต้องทราบเสมอว่าระบบอยู่ในสถานะใด

------------------------------------------------------------------------

# Human-Centered Language

ใช้ภาษาที่คนในร้านเข้าใจ

  หลีกเลี่ยง       ใช้แทน
  ------------- ------------------
  Transaction   กำลังบันทึก
  Exception     เกิดข้อผิดพลาด
  Aggregate     \- (ไม่แสดงต่อผู้ใช้)
  Repository    \- (ไม่แสดงต่อผู้ใช้)

------------------------------------------------------------------------

# UX Checklist

ก่อนปล่อย Feature ใหม่ ให้ตรวจสอบ

-   ผู้ใช้เข้าใจสิ่งที่จะเกิดขึ้นหรือไม่
-   มี Feedback ครบหรือไม่
-   มีข้อความอธิบายเมื่อผิดพลาดหรือไม่
-   ลดจำนวนคลิกได้หรือไม่
-   ใช้งานได้ดีในช่วงเร่งด่วนหรือไม่

------------------------------------------------------------------------

# North Star

> ออกแบบประสบการณ์ให้ผู้ใช้มีสมาธิกับงาน ไม่ใช่กับโปรแกรม

เมื่อผู้ใช้ลืมไปว่ากำลังใช้ระบบ และทำงานได้อย่างต่อเนื่อง นั่นคือ Human Experience ที่ดี
